# ADR-0003: 매장 스코프 — 단일 매장

- **Status**: Accepted
- **Date**: 2026-06-27

## Context

포인트 적립 시스템은 매장 구조에 따라 데이터 모델과 화면이 크게 달라진다.

대안:

- **단일 매장**: 가장 단순. 매장 식별자가 필요 없고, 손님 → 포인트 잔액이 1:1 관계.
- **체인/프랜차이즈**: 여러 매장에서 같은 포인트가 통용. 매장 간 정산, 본사 관리 화면이 추가됨.
- **여러 가게 플랫폼**: 독립된 매장들이 한 플랫폼에 입점. 매장 ID, 매장별 적립 정책, 매장 검색 화면 등이 필요.

이번 프로토타입은 한 카페의 사장님이 직접 사용하는 시나리오를 가정한다.

## Decision

- 매장은 **단일 매장**으로 고정한다.
- 매장 식별자/매장 선택 UI는 만들지 않는다.
- 매장 이름은 상수로 둔다 (`STORE_NAME = '진의 카페'`).
- 적립 정책(적립률) 역시 매장 단위 상수다.

## Consequences

좋은 점:

- 데이터 모델이 매우 단순: `customer(phone) → point_balance`, `point_event(customer, amount, point, created_at)`
- 화면에서 매장 선택/스위칭 UI를 제거할 수 있어, 사장님 적립 화면이 한 손으로 빠르게 끝남
- 손님 화면도 "내 포인트"만 보여주면 됨

나쁜 점 / 따라오는 후속 결정:

- 가게 2호점이 생기는 순간 이 구조로는 대응 불가. 그때는 새 ADR로 매장 다중화 결정을 다시 내려야 한다
- 매장 이름/적립률 등이 코드에 박혀 있어, 변경 시 배포가 필요 — MVP 단계에서 가게 설정 화면을 추가할 수 있음 ([[adr-store-settings]] 후보)
