# ADR-0006: 화면 전환 — 데모용 모드 스위처 (라우터 미도입)

- **Status**: Superseded by [ADR-0010](./0010-react-router.md)
- **Date**: 2026-06-27

## Context

두 번째 화면(손님 — 내 포인트 조회)이 추가되면서, 사장님 화면과 손님 화면을 한 dev 환경에서 어떻게 다 보여줄지 결정해야 한다.

대안:

- **React Router 도입**: 표준적이고 확장성이 높지만, 현 단계에서는 화면이 2~5개 수준이고 URL 디자인 자체가 아직 결정되지 않았다. 라우터를 도입하는 순간 경로/네스팅/가드 같은 결정이 따라온다.
- **해시 라우팅 (수동)**: 가볍지만 손님 vs 사장님 모드는 본질적으로 다른 진입점(다른 도메인, 다른 디바이스)에서 들어와야 하는 화면이라, URL 분기에 매핑하는 것이 어색하다.
- **데모용 토글**: 페이지 상단에 작은 "데모 화면" 스위처를 두고 상태로 전환. 실 제품에서 어떻게 분리될지와 무관하게, 디자인/플로우를 빠르게 비교하는 용도로만 사용.

이번 단계는 [ADR-0001](./0001-tech-stack.md)에서 정의한 대로 **UI 목업**이며, 톤과 흐름을 빠르게 비교하는 것이 목표다. 진짜 제품에서 사장님과 손님은 같은 페이지 안에서 스위치하지 않는다 — 다른 URL/디바이스/계정으로 진입한다.

## Decision

- React Router 등 라우팅 라이브러리는 **이번 단계에서는 도입하지 않는다**.
- 페이지 최상단에 작은 **데모 모드 스위처**를 둔다 (라벨: "데모 화면", 토글: 사장님 / 손님).
- 모드는 `App.tsx`의 `useState`로만 관리한다. URL/쿼리스트링에 노출하지 않는다.
- 화면 코드는 `src/screens/`에 모드별로 분리한다 (`OwnerRewardScreen.tsx`, `CustomerPointScreen.tsx`).
- 공유 유틸은 `src/lib/`로 분리한다 (전화번호 포맷, 상수 등).

> **이 결정은 이후 [ADR-0010](./0010-react-router.md)에서 React Router 도입으로 대체됐습니다.**

## Consequences

좋은 점:

- 라우팅 결정을 미루고 디자인 반복에 집중할 수 있다
- 화면 코드가 모드별로 분리되어 있어, 나중에 실제 라우터를 도입할 때 그대로 라우트 컴포넌트로 매핑 가능
- 디자이너/이해관계자가 두 화면의 톤 일관성을 한 화면에서 비교할 수 있다

나쁜 점 / 따라오는 후속 결정:

- 데모 토글이 실제 제품의 UX가 아니라는 점을 항상 명확히 해야 한다 → 라벨에 "데모 화면"이라고 명시
- 화면이 4~5개를 넘어가는 시점에 정식 라우팅 결정이 필요 ([[adr-routing]] 후보)
- 손님/사장님 인증 경로(다른 URL, 다른 권한)를 분리할 때 별도 ADR이 필요
