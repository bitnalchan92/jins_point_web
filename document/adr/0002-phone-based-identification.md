# ADR-0002: 손님 식별 방식 — 전화번호 only

- **Status**: Accepted
- **Date**: 2026-06-27

## Context

손님이 포인트를 적립/조회할 때 "누구"인지 식별할 방법이 필요하다. 동네 카페/디저트 매장이라는 환경을 고려하면, 손님은 보통 결제대 앞에서 매우 짧은 시간 안에 본인 확인을 해야 한다.

대안:

- **회원가입 + 이메일/비밀번호 로그인**: 데이터는 풍부해지지만, 손님 입장에서 진입장벽이 가장 높다. 카페 카운터 앞에서 비밀번호 입력은 비현실적이다.
- **QR 코드 (개인 발급)**: 본인 확인은 빠르지만 QR을 분실/관리해야 한다. 단골 손님에게는 좋지만 첫 방문 손님에게는 마찰이 크다.
- **전화번호 입력**: 한국에서 가장 익숙한 방식. 별도 가입 절차 없이 사장님이 그 자리에서 입력하면 끝난다. 손님은 자기 번호만 기억하면 된다.

매장이 손님의 매출 데이터/연락처를 활용해 마케팅을 할 가능성도 있지만, 이는 별도 결정 사항으로 다룬다.

## Decision

- 손님 식별자는 **전화번호 한 가지**다.
- 별도 회원가입 절차는 없다. 첫 적립 순간 자동으로 손님 레코드가 생성된다 (사장님 화면에서 "신규 손님" 뱃지로 시각적으로 알린다).
- 비밀번호/PIN 등 본인 인증 절차는 두지 않는다 — 적립은 사장님이 직접 처리하고, 조회는 본인 번호를 아는 사람만 가능하다.
- 표시할 때는 `010-****-5678` 형태로 가운데 4자리를 마스킹한다.

## Consequences

좋은 점:

- 손님 마찰이 0에 가깝다. 첫 방문에서도 적립이 즉시 가능
- 사장님도 별도 가입 안내를 할 필요가 없다
- 단골 손님은 번호만 기억하면 어디서든 본인 포인트 확인 가능

나쁜 점 / 따라오는 후속 결정:

- 전화번호를 아는 제3자가 손님 포인트를 조회할 수 있다 (잔액 노출 정도이며, 차감은 사장님만 가능하므로 위험은 낮음)
- 전화번호는 개인정보다. 저장/마스킹/접근 통제에 대한 별도 결정이 필요 ([[adr-pii-handling]] 후보)
- 한 번 입력한 전화번호를 손님이 바꾸고 싶을 때의 플로우가 필요 — 사장님 화면에서 병합/이전 기능 ([[adr-customer-merge]] 후보)
