# ADR-0009: 손님 조회 — 휴대폰 뒷자리 4자리 (캐셔 UX)

- **Status**: Accepted
- **Date**: 2026-06-29

## Context

사장님이 카운터에서 포인트 적립 시 손님을 빠르게 식별해야 한다.

[ADR-0002](./0002-phone-based-identification.md)에서 손님 식별 기준은 전화번호로 결정됐다. 그러나 실제 POS 환경에서 11자리 전체 번호를 매번 입력하는 것은 속도가 느리고 실수 가능성이 높다.

대안:

- **11자리 전체 입력**: 정확하지만 캐셔 입력 속도가 느리고, 손님이 불러주기도 번거롭다.
- **이름 검색**: 한글 입력 전환이 필요하고, 동명이인 처리가 복잡하다.
- **뒷자리 4자리**: 한국 휴대폰 번호 특성상 뒷자리 4자리만으로 대부분의 손님을 구분할 수 있다. 손님이 불러주기 쉽고 키패드 입력 속도가 빠르다.

## Decision

- 포인트 적립 화면(`OwnerRewardScreen.tsx`)에서 손님 조회는 **뒷번호 4자리**로 한다.
- 4자리 입력 → 매칭 손님 조회 → **3단계 흐름** 적용:
  - STEP 1: 뒷번호 4자리 입력
  - STEP 2: 복수 매칭 시 손님 선택 (0명 또는 1명이면 자동 통과)
  - STEP 3: 결제금액 입력 → 적립 또는 사용
- 전화번호 전체는 `src/lib/data.ts`에 저장하며, 조회는 뒷 4자리 substr로 필터링한다.

## Consequences

좋은 점:

- 캐셔 동선 최소화 — 숫자 키패드만으로 전체 흐름 처리 가능
- 복수 매칭 UI를 갖춰 충돌을 명시적으로 해소
- 손님 입장에서 외우기 쉬운 번호 4자리만 불러주면 됨

나쁜 점 / 따라오는 후속 결정:

- 뒷자리 4자리가 동일한 손님이 많아지면 매칭 목록이 길어질 수 있다 (현 단계에서는 허용)
- 신규 손님은 4자리 조회로 등록할 수 없으므로, 손님 관리 탭에서 별도 추가 흐름이 필요 (현재 구현됨)
